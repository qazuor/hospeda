# SPEC-255 — Airtable Migration & Decommission

## Overview

**Goal.** Import all relevant data currently living in Airtable into the first-party
social-automation Postgres tables introduced by SPEC-254, re-wire the Custom GPT Actions
and Make scenarios to talk exclusively to the new backend, verify parity between Airtable
and the DB, then stop the Airtable runtime dependency permanently.

**Motivation.** SPEC-254 builds the new social automation backend (tables:
`social_campaigns`, `social_audiences`, `social_platforms`, `social_platform_formats`,
`social_hashtag_sets`, `social_hashtags`, `social_post_footers`, `social_content_batches`,
`social_posts`, `social_assets`, `social_post_targets`, `social_publish_logs`). Until data
is migrated, both Airtable and the new DB must run in parallel — a dual-write hazard that
introduces drift and operational overhead. SPEC-255 closes that window by migrating once,
cutting over integrations, confirming parity, and decommissioning Airtable.

**Success criteria.**

1. All catalog data from Airtable (campaigns, audiences, platforms, platform formats,
   hashtag sets, hashtags, post footers, content batches) is present in the SPEC-254 DB
   tables with correct slug/name merge keys; no duplicate rows after repeated importer runs.
2. The Custom GPT Actions call `/api/v1/ai/social/catalog` and `/api/v1/ai/social/drafts`
   instead of the legacy Make webhooks that proxied to Airtable.
3. Make scenarios follow the SPEC-254 PUSH model: the backend enqueues jobs and pushes
   them to a single configurable Make webhook; Make publishes and calls back
   `/api/v1/ai/social/posts/:id/claim` and `/api/v1/ai/social/posts/:id/result`. Make no
   longer reads from or writes to Airtable.
4. A parity report (generated by the importer) shows zero unresolved discrepancies between
   the exported Airtable snapshot and the DB state.
5. All existing Make-to-Airtable connections are disabled in Make; the Airtable base is
   set read-only (archived); no production code path references Airtable.

**Hard dependency.** This spec MUST NOT start implementation until SPEC-254 is merged and
stable on staging. The DB tables, service layer, and API endpoints built by SPEC-254 are
the migration target; nothing here makes sense without them.

## User Stories & Acceptance Criteria

### US-1 — One-off data import runs idempotently from an Airtable export

**As a** platform operator, **I want** to run a single import script against an Airtable
JSON/CSV export, **so that** all catalog and content data is loaded into the SPEC-254 DB
tables without duplicates, and re-running the script produces the same result.

**Acceptance Criteria:**

- **Given** an Airtable export file (JSON or CSV) is present at the configured input path
  and the SPEC-254 tables exist in the target DB,
  **When** the operator runs `pnpm cli airtable:import --file <path>`,
  **Then** every row in the export is upserted into the corresponding SPEC-254 table using
  slug (for catalog tables) or a content-hash key (for posts/assets) as the idempotent
  merge key; the command exits 0 and prints a summary line per table
  (`imported: N, skipped: N, errors: 0`).

- **Given** the import has already been run once,
  **When** the operator runs the same command again with the same export file,
  **Then** no duplicate rows are created; all counts in the summary show `imported: 0,
  skipped: N, errors: 0` for each table.

- **Given** one or more rows in the export are structurally invalid (missing required
  fields, type mismatch),
  **When** the import encounters those rows,
  **Then** each invalid row is written to an error log (`import-errors.jsonl` alongside the
  input file) with the row index, table name, and validation error message; the remaining
  valid rows are still imported; the command exits with code 1 and prints the count of
  invalid rows.

- **Given** the export file does not exist at the specified path,
  **When** the operator runs the import command,
  **Then** the command immediately exits with code 1 and prints `Error: file not found at
  <path>`; no DB writes occur.

### US-2 — Parity report confirms data completeness before cutover

**As a** platform operator, **I want** a parity report that compares the Airtable export
snapshot to the current DB state, **so that** I can confirm zero data loss before disabling
Airtable.

**Acceptance Criteria:**

- **Given** an Airtable export file and a live DB connection,
  **When** the operator runs `pnpm cli airtable:parity --file <path>`,
  **Then** the command prints a per-table summary showing: total Airtable rows, rows found
  in DB, rows missing from DB, rows in DB not in export; and exits 0 if missing count is 0
  for all tables.

- **Given** one or more Airtable rows are missing from the DB,
  **When** the parity check runs,
  **Then** the command exits with code 1, lists each missing row (table, Airtable record ID,
  slug/name), and includes a human-readable hint ("Run `airtable:import` to resolve").

- **Given** the DB contains rows that do not appear in the export (rows created natively
  after import),
  **When** the parity check runs,
  **Then** those rows are listed in the "DB-only" section of the report but do NOT cause a
  non-zero exit code; they are expected and acceptable.

### US-3 — Custom GPT Actions point to the new backend endpoints

**As an** operator using Custom GPT, **I want** the GPT Action schema to reference
`/api/v1/ai/social/catalog` and `/api/v1/ai/social/drafts` instead of the old Make webhook
URLs, **so that** the GPT no longer depends on Airtable data served through Make.

**Acceptance Criteria:**

- **Given** the updated OpenAPI / Actions schema is uploaded to the Custom GPT configuration,
  **When** the GPT calls the catalog action,
  **Then** it receives a JSON response from `/api/v1/ai/social/catalog` (authenticated via
  the existing AI API key) containing the current platforms, hashtag sets, audiences, and
  campaigns from the SPEC-254 tables; the response structure matches what the GPT system
  prompt expects.

- **Given** the updated schema is in place,
  **When** the GPT submits a draft via the drafts action,
  **Then** a new `social_posts` row is created with status `draft` and the GPT receives a
  confirmation payload including the new post ID; no write to Airtable occurs.

- **Given** the legacy Make webhook URL is present anywhere in the Custom GPT configuration
  (action schema, system prompt, or instructions),
  **When** the cutover is complete,
  **Then** the operator has removed all references to `hook.us1.make.com` and any
  Airtable-proxying URLs; the parity checklist item for GPT Actions is ticked.

### US-4 — Make scenarios follow the SPEC-254 PUSH model without Airtable

**As an** operator managing Make automation, **I want** Make scenarios that receive job
pushes from the backend, publish the content, and call back `/claim` and `/result`,
**so that** Make no longer queries or updates Airtable at any step.

**Acceptance Criteria:**

- **Given** the backend pushes a publish job to the configured Make webhook
  (`SOCIAL_MAKE_WEBHOOK_URL`),
  **When** Make receives the job payload,
  **Then** Make publishes to the target platform using the data in the payload (no Airtable
  lookup step); on success, Make calls `POST /api/v1/ai/social/posts/:id/claim` and then
  `POST /api/v1/ai/social/posts/:id/result` with the publish result; the `social_post_targets`
  row transitions to status `published` and `social_publish_logs` gets a new entry.

- **Given** Make encounters a publish error for a job,
  **When** Make calls back `/result`,
  **Then** the payload includes `success: false` and an error message; the
  `social_post_targets` row transitions to status `failed`; the error is recorded in
  `social_publish_logs`; no Airtable record is updated.

- **Given** the Make scenario redesign is complete,
  **When** an operator inspects each Make scenario that previously had an Airtable
  connection module,
  **Then** zero Airtable modules remain; all data reads and writes go through the SPEC-254
  backend endpoints.

### US-5 — Airtable is formally decommissioned after parity is confirmed

**As a** platform operator, **I want** a clear decommission checklist to be executed and
recorded, **so that** the team has an auditable record that Airtable is no longer a
runtime dependency and the base can be safely archived.

**Acceptance Criteria:**

- **Given** the parity report exits 0 and GPT + Make cutover is verified,
  **When** the operator executes the decommission steps,
  **Then** all Make scenario Airtable connections are disabled (not just paused), the
  Airtable API key used by Make is revoked in Airtable's developer settings, and the
  Airtable base is set to read-only / archived state.

- **Given** decommission is complete,
  **When** a search is run across the entire codebase for Airtable API hostnames and the
  legacy Make webhook base URL,
  **Then** zero matches are found in production source files (scripts used only during
  migration are allowed to reference them but must be clearly marked as migration-only
  and not imported by any production entry point).

- **Given** the `AIRTABLE_API_KEY` environment variable exists in any app's env registry
  or `.env.example`,
  **When** decommission is finalized,
  **Then** those references are removed from `packages/config/src/env-registry.*.ts` and
  from all `.env.example` files; CI `env:check:registry` passes after the removal.

## Technical Approach

**Approach.** Three sequential work streams, each gated on the previous:
(1) build and run the importer; (2) redesign Make + Custom GPT; (3) verify parity and
decommission.

### Stream A — One-off import script (`scripts/`)

- **Location:** `scripts/airtable-import/` — a standalone Node.js / TypeScript script
  invoked via `pnpm cli airtable:import` and `pnpm cli airtable:parity`. Marked
  migration-only; not imported by any app entry point.
- **Input:** Airtable JSON export (exported manually from the Airtable UI before cutover)
  or per-table CSV files. The script accepts `--file <path>` for a single JSON bundle
  or `--dir <path>` for a directory of per-table CSVs.
- **Merge keys** (idempotent upsert strategy):
  - `social_campaigns` → `slug`
  - `social_audiences` → `slug`
  - `social_platforms` → `slug`
  - `social_platform_formats` → `(platform_id, slug)`
  - `social_hashtag_sets` → `slug`
  - `social_hashtags` → `(hashtag_set_id, tag)`
  - `social_post_footers` → `slug`
  - `social_content_batches` → `slug`
  - `social_posts` → content hash of `(batch_id, caption_draft)` or Airtable record ID
    stored in a `airtable_record_id` nullable column (added via extras migration if SPEC-254
    did not include it)
  - `social_assets` → file URL hash or Airtable record ID
- **Validation:** each row is validated against the corresponding Zod schema from
  `@repo/schemas` before upsert. Invalid rows go to `import-errors.jsonl`.
- **DB access:** uses `@repo/db` models directly (same as services); runs inside a single
  transaction per table for atomicity.
- **No runtime dependency:** the script is never `require()`-d or `import()`-d by
  `apps/api`, `apps/admin`, or `apps/web`. It is a one-shot migration tool.

### Stream B — Custom GPT Actions schema update

- **Location:** `scripts/airtable-import/gpt-actions-schema.yaml` — the updated OpenAPI
  YAML for the Custom GPT, replacing all `hook.us1.make.com` URLs with the Hospeda API
  base URL.
- **Auth:** the GPT continues to use the existing AI API key header
  (`X-Hospeda-AI-Key: <key>`); no new auth mechanism.
- **Endpoints exposed to GPT:** `GET /api/v1/ai/social/catalog` (returns platforms,
  hashtag sets, campaigns, audiences as a flat catalog for the GPT to reference) and
  `POST /api/v1/ai/social/drafts` (creates a `social_posts` row in draft status). Both
  endpoints are defined in SPEC-254; this spec only updates the consumer (GPT config).
- **No code change in the API** for this stream — purely a configuration update in the
  Custom GPT platform UI plus the committed YAML artifact for auditability.

### Stream B2 — Make scenario redesign

- **PUSH model:** SPEC-254 defines a `social_post_targets` dispatch loop that pushes job
  payloads to `SOCIAL_MAKE_WEBHOOK_URL`. Make receives the payload and publishes.
- **Callback endpoints** (defined in SPEC-254): `POST /claim` sets `claimed_at`;
  `POST /result` records outcome in `social_publish_logs` and transitions status.
- **Make changes** (configuration, not code):
  - Remove all Airtable Search / Create / Update modules from every social automation
    scenario.
  - Replace the trigger (was: Airtable watch → lookup → publish) with: webhook receiver
    (`social_post_targets` payload) → platform-specific publish module → HTTP module calling
    `/claim` → HTTP module calling `/result`.
  - The `SOCIAL_MAKE_WEBHOOK_URL` env var is registered in the API's env registry (added by
    SPEC-254); verify it is set in Coolify for staging and prod.
- **No new API routes** in this spec; all Make callbacks use SPEC-254 endpoints.

### Stream C — Parity check & decommission

- **`airtable:parity` command** (same script package as Stream A): loads the same export
  file, queries the SPEC-254 tables, diffs by merge key, prints the report.
- **Decommission checklist** (executed manually by the operator, recorded in a
  `scripts/airtable-import/DECOMMISSION.md` artifact committed to the repo):
  1. Parity report exits 0.
  2. GPT Actions smoke-tested against new endpoints.
  3. Make scenario smoke-tested end-to-end (push → publish → claim → result).
  4. All Make Airtable modules disabled.
  5. Airtable API key revoked.
  6. Airtable base archived (read-only).
  7. `AIRTABLE_API_KEY` removed from env registry and `.env.example` files.
  8. Codebase search for Airtable hostnames returns zero matches in production source.
  9. `env:check:registry` CI gate passes.

**Key files to create or modify:**

- `scripts/airtable-import/index.ts` — CLI entry point (`airtable:import`, `airtable:parity`)
- `scripts/airtable-import/importers/` — one importer per SPEC-254 table
- `scripts/airtable-import/validators/` — Zod-based row validation (reuses `@repo/schemas`)
- `scripts/airtable-import/parity.ts` — parity diff logic
- `scripts/airtable-import/gpt-actions-schema.yaml` — updated Custom GPT Actions schema
- `scripts/airtable-import/DECOMMISSION.md` — operator checklist (filled in at cutover)
- `packages/db/src/migrations/extras/0NN-social-posts-airtable-record-id.column.sql` — adds
  `airtable_record_id` nullable column to `social_posts` and `social_assets` if SPEC-254
  did not include it (check first; skip if already present)
- `packages/config/src/env-registry.*.ts` — REMOVE `AIRTABLE_API_KEY` entries at
  decommission (Stream C only, not Stream A)

**Dependencies:**

- `@repo/db` — Drizzle models for all SPEC-254 tables
- `@repo/schemas` — Zod schemas for row validation
- `papaparse` or built-in `csv-parse` — CSV parsing (import only; not bundled into any app)
- No new runtime packages added to `apps/`

**Patterns:**

- All DB writes through `@repo/db` models (never raw SQL in the script).
- RO-RO for all importer functions: `importCampaigns({ rows, db }): Promise<ImportResult>`.
- Idempotent upsert via Drizzle `onConflictDoUpdate()` with the merge key columns listed above.
- Error accumulation (never throw on row-level errors; collect and report at end).

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Airtable export is incomplete or uses a schema that diverged from the SPEC-254 model | High | Export all tables before starting import; run `airtable:parity` immediately after import; any divergence must be resolved before cutover |
| Slug collisions: two Airtable records map to the same slug in the DB | Medium | Importer detects slug collision on upsert and logs a conflict entry; operator resolves manually before re-running |
| `social_posts` merge key ambiguity (Airtable has no canonical slug) | Medium | Use Airtable record ID as the merge key stored in `airtable_record_id`; requires adding that nullable column via an extras migration if SPEC-254 omits it |
| Make scenario redesign breaks publish flow during transition | High | Keep old Airtable-based scenario active (but paused) until the new PUSH scenario is smoke-tested end-to-end on staging; only disable old scenario after parity is confirmed |
| Custom GPT system prompt references Airtable field names or Make webhook URLs inline | Medium | Audit the full GPT system prompt before cutover; update any hardcoded field name mappings to match the SPEC-254 catalog response shape |
| `SOCIAL_MAKE_WEBHOOK_URL` not set in Coolify when SPEC-254 deploys | Medium | Check env var presence in staging before running any Make smoke test; the import script itself does not need this var (it is backend-only) |
| Re-running importer on production after data has been edited natively could overwrite newer data | High | Run importer once on a clean DB (or on staging only); never re-run on production after the cutover date; the `DECOMMISSION.md` checklist records the final run timestamp |
| Decommissioning Airtable API key before all Make scenarios are updated leaves scenarios in a broken state | High | Revoke API key ONLY after step 4 of the decommission checklist (all Make Airtable modules disabled and smoke-tested) |
