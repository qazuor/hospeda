# Database Migrations Guide

This is the canonical how-to for schema changes in Hospeda. For the architectural rationale see
[ADR-029](../decisions/ADR-029-versioned-migration-strategy.md). For Drizzle-invisible objects
and the extras script see [ADR-017](../decisions/ADR-017-postgres-specific-features.md).

---

## Two carriles, one rule

Schema changes live in exactly one of two directories:

| Carril | Directory | Contents | Applied |
|--------|-----------|----------|---------|
| Structural | `packages/db/src/migrations/` | Drizzle-generated .sql files (tables, columns, indexes, FKs, enums) + hand-edited data conversions | Once, in order, by `drizzle-kit migrate` |
| Extras | `packages/db/src/migrations/extras/` | Hand-written idempotent SQL for Drizzle-invisible objects (triggers, matviews, CHECK, special indexes) | Every time, by `db:apply-extras` |

**Golden rule:**

- Can Drizzle declare it in the TS schema? -> `migrations/` (run `db:generate`).
- Is it a trigger, matview, CHECK with function calls, or special index? -> `migrations/extras/`.
- Is it a data conversion (type-cast USING / row UPDATE)? -> `migrations/`, hand-edit the `USING`.

## Day-to-day: local dev

Local dev uses `push` — it's fast, disposable, and requires no journal management.

```bash
# Normal dev loop
pnpm db:fresh-dev      # Drop + push schema + seed (start here every day)
pnpm db:push           # Apply TS schema changes to local DB without a migration file
pnpm db:apply-extras   # Re-apply triggers, matviews, CHECK constraints after push
```

`push` NEVER runs against the VPS. If you find yourself thinking about it, stop.

## Before opening a schema PR

Run these steps every time you change `packages/db/src/schemas/**`:

```bash
# 1. Generate a migration file from the current schema diff
pnpm --filter @repo/db db:generate

# 2. Review the generated .sql file in packages/db/src/migrations/
#    If a column type change needs a data conversion, hand-add the USING:
#    ALTER TABLE ... ALTER COLUMN ... TYPE new_type USING col::text::new_type;

# 3. Commit the .sql file and the updated meta/_journal.json together
git add packages/db/src/migrations/NNNN_*.sql
git add packages/db/src/migrations/meta/_journal.json
git commit -m "chore(db): generate migration for <what changed>"
```

The **drift guard** (`scripts/check-schema-drift.sh`) runs in CI and fails if the TS schema has
diverged from the committed migrations. A missing migration blocks the PR — it is not optional.

### Data conversions

Drizzle generates `ALTER COLUMN ... TYPE` without a `USING` clause. If PostgreSQL cannot coerce
the value automatically (e.g. `text` -> custom enum), the migrate step will fail at runtime.

Hand-edit the generated file to add the explicit conversion:

```sql
-- Generated by drizzle-kit (partial):
ALTER TABLE "posts" ALTER COLUMN "status" TYPE post_status_enum;

-- After your edit:
ALTER TABLE "posts" ALTER COLUMN "status" TYPE post_status_enum
  USING status::text::post_status_enum;
```

Commit the edited file. The `USING` is now version-controlled and will run on every VPS.

## Adding a Drizzle-invisible object

Triggers, materialized views, CHECK constraints with function calls, and special indexes go in
`packages/db/src/migrations/extras/`.

Naming convention: `NNN-name.kind.sql` where:

- `NNN` is a 3-digit zero-padded number that determines execution order
- `name` is a short slug describing the object
- `kind` is one of `trigger`, `matview`, `constraints`, `indexes`

Example: `010-audit-log.trigger.sql`

Requirements:

- The file must be **idempotent** — use `CREATE OR REPLACE` for functions and `IF NOT EXISTS`
  for indexes and constraints. The script is re-run on every `db:apply-extras`.
- Do not put DML (INSERT/UPDATE/DELETE) in extras files — those belong in seed scripts.

```bash
# After adding the file, test it locally
pnpm db:apply-extras
# Verify the object exists
psql $HOSPEDA_DATABASE_URL -c "\df refresh_search_index"  # example
```

## Applying migrations on the VPS

Use the `hops` VPS tooling — never run drizzle commands directly on the VPS.

```bash
# Staging
hops db-migrate --target=staging

# Production
hops db-migrate --target=prod
```

`hops db-migrate` runs:

1. `pg_dump` backup of the target database.
2. `drizzle-kit migrate` (applies all pending `migrations/*.sql`).
3. `db:apply-extras` (re-applies all `migrations/extras/*.sql`).

The command aborts on any failure. The backup is there so a human can restore if something
goes wrong.

### Pre-migration column snapshot for lossy data migrations

`hops db-migrate` already takes a full `pg_dump` before applying. That dump is the complete
safety net, but restoring a SINGLE column from a full prod dump is slow and error-prone. When
a pending migration performs a **lossy in-place data transform** (it overwrites a column and
the original cannot be reconstructed from the new value), take an additional lightweight
per-column snapshot right before promoting, so a targeted restore is a one-liner.

**SPEC-187 / `0008_strip_accommodation_description_markdown.sql`** is the canonical case: it
strips markdown from `accommodations.description` in place and the original markdown is
unrecoverable. `0011_restrip_accommodation_description_markdown.sql` re-cleans what 0008 left
behind (surviving `_emphasis_`, orphan `!` from images) but is also one-way. Before the first
`staging → main` promotion that carries these migrations, snapshot the column on **prod**:

```sql
-- Run on prod BEFORE `hops db-migrate --target=prod` (i.e. before 0008 runs)
CREATE TABLE IF NOT EXISTS accommodations_description_backup AS
SELECT id, description, updated_at FROM accommodations;
```

If the strip corrupts any row, restore just that column (whole-column or filtered by id):

```sql
UPDATE accommodations a
SET    description = b.description
FROM   accommodations_description_backup b
WHERE  a.id = b.id;
```

Keep the table for at least one release cycle. Once a human has confirmed the public
accommodation pages render correctly (no leftover `**`/`_`/orphan `!`), drop it:

```sql
DROP TABLE accommodations_description_backup;
```

### Reset (wipe and rebuild)

The `--reset` flag drops the schema and rebuilds from scratch. **Destructive — use with care.**

```bash
hops db-migrate --target=staging --reset   # Wipes staging, prompts for confirmation
hops db-migrate --target=prod --reset      # Wipes prod, requires typed confirmation
```

After a reset, extensions (`pgcrypto`, `uuid-ossp`, `unaccent`) are NOT present because
`DROP SCHEMA public CASCADE` removes them. `hops db-migrate --reset` handles this automatically
by applying `packages/db/scripts/reset/001-extensions.sql` before `migrate`. If you ever
run the reset SQL manually, you must apply `001-extensions.sql` yourself first.

## Seeding after migrate

Seeding is separate from migrations:

```bash
hops db-seed --target=staging        # Seed staging (no migrate)
hops db-seed --target=staging --migrate  # Migrate then seed in one step
```

## Testing

Three shell tests cover migration correctness:

| Test | File | What it checks |
|------|------|----------------|
| Round-trip | `packages/db/test/migrations/t015-round-trip.sh` | Apply baseline + all migrations on a fresh DB |
| Idempotency | `packages/db/test/migrations/t016-idempotency.sh` | Re-run `db:apply-extras` twice, confirm no errors |
| Conversion regression | `packages/db/test/migrations/t017-conversion-regression.sh` | Known USING conversions produce correct row data |

Run them locally against a throwaway DB before opening a migration PR:

```bash
HOSPEDA_TEST_URL="postgresql://user:pass@localhost:5432/hospeda_migration_test" \
  bash packages/db/test/migrations/t015-round-trip.sh
```

## Common mistakes

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| Ran `db:push` and forgot `db:generate` | Drift guard blocks CI | Run `db:generate`, commit the file |
| Changed a column type without `USING` | `migrate` fails at runtime | Hand-edit the `.sql` to add `USING` |
| Put a trigger in `migrations/` instead of `extras/` | Trigger not re-applied after reset | Move to `extras/`, make it idempotent |
| Ran `push` against a VPS | Journal desynced, rollback impossible | Restore from the `pg_dump` backup |
| Forgot `db:apply-extras` after `migrate` | Triggers/matview missing | Run `pnpm db:apply-extras` |
